## API Route Analysis

### Route Structure

10 API routes organized by feature:
- `/bookings/` - Booking CRUD (create, cancel, reschedule)
- `/calendars/` - Google Calendar sync
- `/cron/reminders` - Scheduled reminder job
- `/slots/` - Available time slots
- `/stripe/` - Payment processing (checkout, webhook, status)
- `/track-click/` - Link analytics

### Strengths

1. **Route Organization** - Well-organized by feature domain
2. **Async/Await** - Modern async patterns throughout
3. **HTTP Methods** - Semantically correct GET/POST usage
4. **Timezone Handling** - Consistent use of date-fns-tz
5. **Webhook Security** - Stripe signature verification
6. **Cron Security** - Bearer token verification

### Issues Identified

| Issue | Severity | Details |
|-------|----------|---------|
| No Zod validation | HIGH | Manual validation only, despite Zod in tech stack |
| `as any` type casts | HIGH | 33+ instances across 6 route files |
| Duplicated parseTimeString | MEDIUM | Defined in both bookings/route.ts and stripe/webhook/route.ts |
| Inconsistent response formats | MEDIUM | Mix of { success, data, error } patterns |
| Generic error messages | MEDIUM | "Internal server error" with no debugging info |
| Auth inconsistency | LOW | Most routes use service role, only /calendars uses user auth |

## Data Flow Analysis

### Supabase Usage Patterns

1. **Server vs Client** - Good separation
   - Server components use `createClient()` from server.ts
   - API routes use `createServiceClient()` for admin ops
   - Client components use browser client appropriately

2. **Type Safety Issues**
   - 58 total `as any` casts across 19 files
   - Root cause: Supabase client insert/update/delete return types difficult to narrow
   - Workaround pattern: `(supabase as any).from("table").insert(...)`

3. **Query Patterns**
   - Good use of relationship queries: `.select(\`*, availability (*)\`)`
   - Some N+1 issues in cron job (email_templates per booking)
   - Sequential queries that could use relations

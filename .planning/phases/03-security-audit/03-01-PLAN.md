## Authentication Security Audit

### CRITICAL: Open Redirect Vulnerability

**Location:** `src/app/auth/callback/route.ts:8`, `src/lib/supabase/middleware.ts:61`

**Issue:** The `redirectTo` parameter is accepted without validation, allowing attackers to redirect users to malicious sites after authentication.

**Attack Vector:**
```
https://app.com/login?redirectTo=//evil.com/phishing
```

**Fix Required:** Validate redirectTo against allowlist of internal paths.

### HIGH: IDOR in Booking Operations

**Location:**
- `src/app/api/bookings/[uid]/cancel/route.ts`
- `src/app/api/bookings/[uid]/reschedule/route.ts`

**Issue:** These endpoints use `createServiceClient()` which bypasses RLS, and perform NO authorization check. Anyone with a booking UID can cancel/reschedule any booking.

**Fix Required:** Add authorization verification (guest email token or host authentication).

### MEDIUM: Missing CSRF Protection

**Issue:** No CSRF tokens on state-changing endpoints.

**Fix:** Implement double-submit cookie pattern or use Next.js Server Actions.

### POSITIVE: Good Practices Found

- Supabase Auth with proper session management
- OAuth tokens stored in database (not localStorage)
- RLS enabled on tables
- Stripe webhook signature verification
